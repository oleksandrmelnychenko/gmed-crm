# Current-State Gap Audit (UA)

> Робочий аудит стану коду станом на **2026-04-15**. Цей файл не змінює канонічні вимоги з `docs/requirements/*`; він фіксує, що вже є в репозиторії, а що ще лишається gap між PDF/Excel scope і реалізацією.

## 1. Що підтверджено як факт

- Excel-файл `1 (Update 2) User Story Salesforce.xlsx` **є** в репозиторії: `docs/1 (Update 2) User Story Salesforce.xlsx`.
- `docs/requirements/01_process-mapping_ua.md` і `docs/requirements/02_anamnese-flow_ua.md` оформлені як канонічні текстові нормалізації відповідних PDF; при розбіжності пріоритет має PDF.
- `docs/testing/user-stories-excel-backlog-audit_ua.md` підтверджує **1:1** трасованість аркуша `User Stories` до `docs/requirements/03_product-backlog_ua.md` по всіх EPIC.
- У billing slice вже є не тільки handoff, а й `quotes`, `invoices` і `Mahnwesen`.
- Портал пацієнта вже існує як окремий current-state slice: документи, privacy requests, invoices, appointment requests, services.
- У clinical case slice вже є section-level version log через `case_versions`; тому versioning не відсутній повністю.
- Identity/session security already має explicit regression coverage: `logout` і `logout-all` одразу denylist-ять поточний access token, revoke-ять refresh-family і гасять інші відкриті сесії того ж користувача без очікування access-token expiry.
- Freeze verification по поточному зрізу зелений: `cargo test --workspace`, `frontend npm test`, `frontend npm run lint`, `frontend npm run build`, `frontend npm run test:e2e` (`22/22`) і consolidated live DB-backed Playwright suite `frontend npm run test:e2e:live` (`47/47`) пройшли без падінь.

## 2. Що виявилось неточним у попередніх зведеннях

- Неправильно писати, що Excel-файл відсутній у `docs/`.
- Неправильно писати, що `Mahnwesen` або `invoices` ще не реалізовані як backend module.
- Неправильно описувати patient portal як повністю відсутній.
- Неправильно описувати versioning анамнезу як повністю відсутній; коректніше казати, що потрібно ще оцінити повноту retention / immutability / audit semantics.

## 3. Підтверджені current-state gaps

### 3.1 Process map / lead gates

- У `leads` і `orders` уже є bounded lifecycle engine: sequential transitions, history events і blockers для `qualification/conversion`, existing-customer `re-check`, `execution`, `closure`, `follow-up`, але це ще не весь process-map scope по всіх суміжних фінансових та операційних вузлах.
- `Schuldenmanagement` тепер оформлений як окремий order-level workflow з queue, owner/review timestamps, `payment_plan / awaiting_payment / escalated / cleared` статусами, блокуванням execution і відображенням у patient re-check; незакритими лишаються зовнішні payment-provider / accounting handoff сценарії, а не базовий debt workflow.
- Гейт `Freigabe Abrechnung` і гілка `Paketleistung` тепер винесені в order process gates, але не покривають увесь фінансовий сценарій кінця-в-кінець.
- Controlled failed-lead archive/delete flow тепер існує як окремий policy-driven workflow з reason/note, audit trail, lifecycle history і delete-as-anonymize semantics замість прямого hard delete.

### 3.2 Anamnese / clinical intake

- Patient profile тепер already має окремий `clinical_warnings` field і time-series `patient_vital_measurements` з `measured_at`, `bp_systolic`, `bp_diastolic`, `heart_rate`, `weight_kg`, `height_cm`, `bmi`, `notes`, `recorded_by`; assignment-scoped `GET/POST /patients/{id}/vitals` уже покриті regression tests.
- Patient profile тепер already має окремий `patient_card_entries` stream для `Karteikartenzeilen`: категоризовані longitudinal clinical entries з `entry_date`, `category`, `source`, `author` і автоматичним входом у загальний patient timeline.
- Patient profile тепер already має окремий `patient_medical_orders` slice для `ärztliche/therapeutische Anordnungen`: structured order type, title, instructions, due date, source, lifecycle `active/completed/cancelled` і timeline integration.
- Patient profile тепер already має окремий `patient_risk_scores` slice для patient-level structured risk scoring: `score_type`, `score_value`, optional `scale_max`, `inputs JSONB`, interpretation/source і append-only timeline integration.
- Case workspace тепер already має окрему `case_text_snippets` library для anamnesis authoring: reusable text blocks with `{patient_name}`, `{patient_pid}`, `{case_id}`, `{case_uuid}`, `{hauptanfragegrund}`, `{zuweiser}`, `{today}` placeholders, PM/CEO snippet management і direct insertion into `aktuelle_anamnese` from the overview panel.
- Case medication slice тепер already підтримує `expiry_date` для `medikamente`, автоматичний expiry scheduler з дедупом через `medication_expiry_events`, in-app confirmation notifications і explicit PM/CEO review confirmation path прямо з case detail.
- `Cardiology`, `Gastroenterology`, `Orthopedics`, `Neurology`, `Pulmonology` і `Urology` тепер виділені як окремі section-level sub-flows з structured assessment, recommendation trigger від symptoms і окремим version log; далі це вже розширення specialty library, а не незакритий базовий gap самого specialty-subflow pattern.
- Appointment cycle тепер already має explicit `care_path_kind` semantics поверх загальних appointments і portal appointment requests: `regular / preventive / control / followup` з backend validation, staff filter/update flow, patient request history і convert-time propagation в scheduled visit.

### 3.3 Billing / finance

- `DATEV` export не підтверджений як реалізований current-state module.
- `E-Rechnung` не підтверджена як реалізований current-state module.
- Немає real payment-provider checkout / settlement confirmation beyond payment-proof handoff.
- Billing slice тепер already має внутрішній cash-based `accounting_entries` ledger / EÜR read model: paid customer invoices матеріалізують `service_revenue` та `cost_passthrough_revenue`, paid `external_invoices` дають `provider_expense`, а `ceo / ceo_assistant / billing` мають read/export доступ до `/invoices/accounting-ledger`; незакритими лишаються зовнішні accounting handoff'и на кшталт `DATEV`, а не сам базовий internal ledger.
- Billing/order slice тепер already має окремий `external_invoices` registry зі статусами `expected/received/approved/paid/overdue/cancelled`, order-bound UI, deadline scheduler і overdue notifications для billing queue; незакритими лишаються зовнішній accounting/payment handoff і settlement integrations, а не сам inbound external-invoice tracking.
- Orders/billing slice тепер already вміє auto-link-ити `cost_passthrough` `order_leistungen` до supporting documents: `external_document_id` або explicitly вибирається, або автоматично резолвиться, якщо в order є рівно один активний receipt/provider-invoice candidate; цей link проходить у quote/invoice line items і surfaced назад як `supporting_documents` в invoice detail.
- `Freigabe Abrechnung` тепер блокує `quote -> invoice` на backend-рівні, а `package coverage / debt hold` уже враховуються в order execution gates; незакритими лишаються зовнішні accounting/payment сценарії, а не сам базовий gate.

### 3.4 Security / analytics / learning / AI

- `eIDAS / QES` e-signature не підтверджена як готовий current-state flow.
- secure in-portal messaging тепер already має client-side `E2E` і для text messages, і для file attachments через per-user message keys та envelope storage на backend; ручний encrypted key backup / import закриває current-state device migration semantics без серверного доступу до private keys.
- `admin/security` тепер already має окремий audit-analytics slice для `it_admin`: failed/blocked login counters, refresh-token theft signal, executive-sensitive access signal, off-hours sensitive-read signal, recent suspicious events stream і top sensitive readers за останні 24h поверх append-only `audit_log`; це вже підтверджено не тільки backend route gating, а й live browser shell proof, де `it_admin` реально відкриває `/admin/security` і бачить весь analytics surface.
- CEO dashboard, окремий reports workspace, role-scoped risk-analysis і forecasting workspace уже покривають current-state метрики по виручці, дебіторці, країнах пацієнтів, клініках, service-type mix, sales-safe medical provider performance/revenue view, PM/interpreter/concierge load, clinic volume, NPS/feedback surfacing, doctor drill-down/export, provider-quality signals по treatment / doctor communication / follow-up completion, organization / service / ambience / value scores, treatment-success / complication rates, written-findings turnaround, response-time KPI signals із appointment communications, medical provider cost intelligence по historical unit-price movement, non-medical provider reporting/search по service portfolio і concierge load, explicit billing KPI scorecard (`invoices / 30d`, `service -> invoice`, on-time `14d`, receivables, dunning, self-pay share, cost passthrough share), sales KPI scorecard (`new/qualified/converted leads`, lead-to-patient conversion, lead geography, new partner clinics), quote pipeline, collections outlook, follow-up milestone pressure і clinic capacity next 30 days; незакритими лишаються predictive / AI-style forecasting і зовнішній accounting intelligence, а не сам базовий executive forecast layer.
- Executive analytics shell тепер має і canonical live/browser proof: `CEO` та `CEO Assistant` окремо відкривають dashboard і reports workspace, а browser рівнем підтверджені `CEO read model`, `Patient manager risk analysis`, `Billing risk analysis`, `Billing KPI scorecard`, `Sales KPI scorecard`, `Forecast pipeline`, `Collections forecast`, `Follow-up forecast`, `Country report` і `Doctor drill-down`; `CEO Assistant` у цьому ж read-only shell вже не тільки бачить executive reports, а й реально качає `clinic-report.csv` та `doctor-report.csv`; окремо `sales` уже browser-рівнем підтверджено на sales-safe reports shell, де видимі `Sales-KPI-Scorecard`, medical-provider performance і country reporting, `medical-provider-report.csv` та `country-report.csv` реально скачуються, а executive-only sections на кшталт billing KPIs, collections, clinic report і doctor drill-down приховані; додатково `patient_manager` і `billing` тепер мають explicit browser proof на role-scoped analytics shell, де `patient_manager` бачить власну KPI-карту й operational forecasting без executive фінансових секцій, а `billing` бачить лише billing-risk і billing/collections analytics без patient-manager чи sales sections, зате може реально скачати `doctor-report.csv` і `provider-cost-report.csv` із дозволених billing-safe sections.
- Provider registry/reference-data shell тепер теж має окремий canonical live/browser proof: `patient_manager` відкриває `/providers`, бачить seeded clinic card у registry, provider sheet із `Provider profile / Service catalog / Linked patients / Interaction history` і повний detail route `/providers/:id` з `Doctors / Services / Templates / Linked patients` tabs; окремо `sales` тепер browser-рівнем підтверджено як read-only consumer цього ж registry без create/save/delete mutation controls, включно з read-only `Templates` tab без `New template` / `Save template` actions і без `Edit/Delete` affordances у `Doctors` / `Services` tabs.
- High-risk `RBAC` boundaries вже підтверджені regression tests не тільки для sales, а й для `CEO Assistant` patient read-only scope плюс reports/forecasting/risk access, `teamlead_interpreter` assignment visibility, operational-role deny на broad analytics workspaces, `it_admin` deny на patient/case/reports workspaces, `billing` deny на medical case detail, а також commercial workspace alignment, де `CEO` має full contracts/quotes access, `CEO Assistant` лишається read-only, `CEO Assistant` так само має read-only invoice/PDF/dunning visibility, `patient_manager` у browser shell може інспектувати invoice detail, але не може міняти billing status чи запускати dunning, `billing` у browser-level order shell тепер має фінансові `process gates` і `external_invoices` mutate-path, але lifecycle/planning/workflow mutations там не рендеряться, а execution/follow-up controls переведені в disabled read-only state, і `sales`/`concierge` не заходять у patient-bound contracts/quotes/invoice routes; окремий live shell pass тепер ще й явно цементує browser-level denied-route normalization для `sales -> /documents`, `sales -> /contracts`, `concierge -> /invoices`, `billing -> /cases` і `it_admin -> /patients|/cases|/reports|/documents`. Залишок тут — це вже exhaustive matrix hardening, а не відсутність базових policy boundaries.
- Lead conversion gate тепер підтверджений не тільки route-level readiness checks, а й browser-level staff smoke: `patient_manager` уже на cards бачить disabled `Convert` для `qualified` lead з `conversion_ready=false` і окремо enabled state для ready lead, без blind POST/422 round-trip; окремо `sales` тепер browser-рівнем підтверджено як intake-only role у цьому ж shell: може створити lead і довести його до `qualified`, але `Convert` на card не рендериться навіть після кваліфікації, тобто patient creation лишається тільки у `patient_manager`.
- Appointment workspace теж вирівняний під ту саму матрицю: `sales`, `billing`, `ceo_assistant` і `it_admin` не відкривають `/appointments` навіть якщо дійдуть до route вручну, а staff navigation більше не світить цей surface поза operational roles; окремо browser-рівнем тепер підтверджено split shell для `teamlead_interpreter`, де assigned teamlead може відповісти на interpreter assignment і reassignment flow лишається видимим, але status/checklist/reminder-creation mutations у тому ж detail workspace не з’являються.
- Feedback workspace/nav тепер теж вирівняний під RBAC matrix: `billing`, `sales`, `interpreter` і `it_admin` не відкривають staff feedback queue/summary, а shell navigation показує цей surface тільки реальним feedback roles; окремо live browser proof уже цементує scoped row visibility для `teamlead_interpreter` і `concierge`, де кожен бачить тільки свій релевантний feedback slice even within the same assigned patient context.
- AI / pseudonymization flow не підтверджений як реалізований current-state slice; якщо AI свідомо виключений з поточного delivery scope, це не треба трактувати як активний runtime gap для поточної хвилі робіт.

### 3.5 Corrected residual list

- Якщо AI входить у scope, реально незакритим внутрішнім product slice після останніх хвиль лишається `AI / pseudonymization -> AI handoff`.
- Якщо AI і зовнішні інтеграції свідомо не беремо, нових внутрішніх `product gaps` на цьому current-state зрізі вже не лишається; більшість колишніх provider / billing / clinical gap-пунктів уже current-state.
- Potential domain extension, але не blocker: richer preventive/control program model, якщо source scope колись вимагатиме окрему preventive orchestration semantics поверх current `care_path_kind` layer.
- Решта незакритого current-state — це вже переважно `verification density / exhaustive RBAC matrix hardening / commit hygiene`, а не відсутність core runtime modules.

## 4. Що вже ближче до реалізації

- Реєстр клінік і лікарів та зв'язка з пацієнтами через appointments/orders, включно з `legal_name / tax_id` на provider-рівні, `languages / licensing` на doctor-рівні, aggregated `linked_patients + interactions` у provider detail і provider-specific document templates (`provider_templates`) для clinic-level generation flow; окремо partner prep templates уже вміють auto-send на `medical appointment -> confirmed`, з portal release і dedupe по appointment/template.
- Patient profile з tabs, timeline, compliance trail, functional labels, consent register і privacy workflow.
