### 1.1 Критичні

#### `bundle-dynamic-imports` — eager bundle усіх секцій кейс-воркспейсу
**Файл:** [frontend/src/pages/case-workspace/index.tsx:10](frontend/src/pages/case-workspace/index.tsx#L10)

14 секційних компонентів імпортуються статично, але в `renderSection` одночасно рендериться лише одна. Користувач зазвичай відкриває 1–2 секції на кейс, але платить завантаженням усіх 14.

**Фікс:**
```tsx
const CardiologySection = React.lazy(() =>
  import("./cardiology-section").then((m) => ({ default: m.CardiologySection }))
);
// ... те ж саме для решти 13 секцій
// рендерити всередині <Suspense fallback={<TabLoader />}>
```

### 1.3 Середні

#### `rerender-memo` — dead `useMemo` перед `useState`
**Файл:** [frontend/src/pages/case-workspace/specialty-section.tsx:71](frontend/src/pages/case-workspace/specialty-section.tsx#L71)

`hydrate = useMemo(...)` обчислюється щорендер, але `useState(() => hydrate)` зчитує init тільки один раз. Батько вже форсує remount через `key={revisionKey}`.

**Фікс:** Прибрати `useMemo`; інлайн `useState<T>(() => ({ ...blankValue, ...(rawValue ?? {}) }) as T)`.

---

#### `rerender-memo-with-default-value` — non-primitive default як prop
**Файл:** [frontend/src/pages/case-workspace/case-item-list.tsx:74](frontend/src/pages/case-workspace/case-item-list.tsx#L74)

`CaseItemList` приймає `blankItem` прямо в `useState(blankItem)`. Усі поточні кол-сайти підіймають `const BLANK = {...}` у module scope, але API провокує баг при inline об'єктах.

**Фікс:** `useState<T>(() => blankItem)` + документувати контракт "стабільний референс", або приймати `createBlankItem: () => T`.

---

#### `rerender-split-combined-hooks` — моно-контекст з 20+ залежностями

**Фікс:** Розділити на два контексти —
- `CaseWorkspaceActionsContext` (caseId + save callbacks + reload, deps `[caseId]`)

Секції, яким потрібні лише actions, перестануть ре-рендеритись на data-churn.

## 2. Backend — ядро (Rust workspace)

### 2.1 Архітектура коротко
Axum + PostgreSQL (sqlx 0.8), воркспейс із 4 крейтів:
- `crates/domain` — типи (Role, Access, Error), domain-моделі
- `crates/shared` — DTO, responses
- `crates/db` — пул + runner міграцій
- `crates/server` — HTTP + middleware + crypto + audit

Точка входу: [crates/server/src/main.rs:8](crates/server/src/main.rs#L8)

### 2.2 Request flow
1. Router ([crates/server/src/lib.rs:34](crates/server/src/lib.rs#L34)) → health/public обходить auth
2. `require_auth` ([crates/server/src/auth/middleware.rs](crates/server/src/auth/middleware.rs)) валідує JWT, кладе `AuthUser` в extensions
3. Audit middleware ([crates/server/src/audit.rs:1](crates/server/src/audit.rs#L1)) логує метадані
4. Handler → `require_any_role()` / `require_exact_role()` → sqlx-запити
5. `IntoResponse` → глобальні security headers ([crates/server/src/security_headers.rs:45](crates/server/src/security_headers.rs#L45))

### 2.3 Сильні сторони (зберегти / не чіпати)
- **Audit-тригер** забороняє UPDATE/DELETE на `audit_log` — ISO 27001 A.8.15 tamper-evidence без app-layer ([crates/server/src/audit.rs:27](crates/server/src/audit.rs#L27))
- **Security headers baseline** — HSTS/CSP/X-Frame на кожній відповіді, включно з 429 ([crates/server/src/security_headers.rs:44](crates/server/src/security_headers.rs#L44))
- **Rate limiting** per-IP, окремі бакети на `/auth/*` ([crates/server/src/lib.rs:25](crates/server/src/lib.rs#L25))
- **Ротація ключів шифрування** + фоновий re-wrap кожні 10 хв ([crates/server/src/main.rs:55](crates/server/src/main.rs#L55))
- **GDPR lead purger** — анонімізація архівних лідів після 180 днів, Art. 5(1)(e) ([crates/server/src/main.rs:159](crates/server/src/main.rs#L159))

#### R2 — `TODO(audit-migrate)` у 9 місцях
**Приклад:** [crates/server/src/routes/contracts.rs](crates/server/src/routes/contracts.rs)

**Чому важливо:** handler тримає транзакцію, а audit middleware бачить статус тільки після response. При rollback у БД audit-запис може вже бути зафіксований невірно, або навпаки — загубитись.

**Рекомендація:** Запровадити уніфікований патерн tx-aware audit:
- Передавати `&mut Transaction` у audit-репозиторій
- Гарантувати, що audit commit/rollback відбувається разом з основною tx
- Замість middleware post-hook — явний `audit.record(&mut tx, event)` всередині handler

#### R3 — `panic!` у `Config::from_env`
**Файл:** [crates/server/src/config.rs:30](crates/server/src/config.rs#L30)

**Чому важливо:** k8s/docker liveness probe не отримує структурованого фідбеку, контейнер просто падає.

**Рекомендація:** Замінити на `Result<Config, ConfigError>` з варіантами `WeakJwtSecret`, `PlaceholderValue`, і дати `main` вивести помилку у структурованому JSON-логу перед exit(1).

#### R4 — Audit channel (cap 512) тихо дропає
**Файл:** [crates/server/src/audit.rs:38](crates/server/src/audit.rs#L38)

**Чому важливо:** при сплеску (напр. масовий імпорт) події губляться — прийнятно по DPIA, але немає алертингу.

**Рекомендація:**
- Додати counter `audit_events_dropped_total` (Prometheus)
- Alert-правило: > 0 за 5 хв = WARN, > 100 = CRIT
- Опціонально: fallback-спулер у файл при переповненні
