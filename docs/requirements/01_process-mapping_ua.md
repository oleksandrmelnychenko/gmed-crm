# Бізнес-процес роботи з пацієнтом

> **Джерело правди (графічне):** `docs/Process Mapping (Kundenjourney allg.)(in Bearbeitung).pdf`. Цей файл — канонічний текстовий опис того ж процесу; при розбіжності з діаграмою пріоритет має PDF. Огляд ієрархії джерел: `docs/00_source-of-truth_ua.md`.

## 1. Призначення документа

Цей документ описує цільовий наскрізний процес:

`Lead / Customer -> Qualification -> Order -> Treatment Program -> Execution -> Billing -> Follow-up`

Він є канонічним процесним описом для:

- product requirements;
- workflow design;
- task/checklist automation;
- architecture alignment.

## 2. Учасники процесу

- `Sales / Vertrieb`
- `Patient Manager`
- `Concierge`
- `Billing / Abrechnung`
- `Interpreter / Dolmetscher`
- `Customer / Lead`

## 3. Основні фази процесу

### Фаза 1. Первинний контакт

1. Клієнт або лід звертається до компанії.
2. Система або співробітник визначає тип звернення:
   - `Lead`
   - `Existing Customer`
3. Для обох сценаріїв виконується первинний контакт із відділом продажу або відповідальним менеджером.

### Фаза 2A. Кваліфікація ліда

1. Проводиться перша кваліфікаційна розмова.
2. Збираються базові дані, потреби, очікування, медичний і немедичний контекст.
3. Запускається compliance-процес:
   - збір контактів;
   - збір посвідчення особи;
   - надсилання та підписання compliance-документів.
4. Якщо compliance не завершений:
   - кейс повертається на уточнення;
   - відкриті питання мають бути закриті до переходу далі.
5. Після завершення compliance:
   - виконується детальна анамнеза;
   - збираються документи;
   - оцінюється кейс;
   - формується початковий пакет послуг.
6. Далі готуються:
   - договір;
   - перше замовлення;
   - кошторис.
7. Проводиться перевірка:
   - чи кваліфікований lead;
   - чи підписані договір і замовлення;
   - чи оплачено кошторис, якщо це потрібно.
8. Якщо умови не виконані, дані підлягають архівації або видаленню згідно політики.
9. Якщо умови виконані, lead конвертується в customer.

### Фаза 2B. Робота з існуючим клієнтом

1. Оновлюються потреби клієнта:
   - медичні;
   - немедичні;
   - дані та документи;
   - anamnesis update.
2. Перевіряється валідність:
   - базових даних;
   - compliance-документів;
   - договірних документів;
   - ідентифікаційних документів.
3. Якщо дані невалідні або неповні, виконується update і повторна перевірка.
4. Перевіряється наявність боргів:
   - якщо є борг, кейс переходить у debt-management сценарій;
   - якщо боргу немає, можна створювати нове замовлення.
5. Для нового замовлення:
   - потрібна freigabe від Abrechnung;
   - якщо Abrechnung не дає freigabe, перевіряється чи **замовлення покривається існуючим пакетом послуг** (Paketleistung);
   - якщо покривається — замовлення проходить без окремого кошторису;
   - якщо не покривається — запускаються стандартні договірні та фінансові перевірки (Auftrag + Kostenvoranschlag → підпис → оплата).

### Фаза 3. Конвертація та запуск кейсу

1. Після успішної кваліфікації або створення замовлення:
   - пацієнту призначається `Patient Manager`;
   - створюється активний робочий кейс.
2. Формується:
   - план обстеження / лікування;
   - кошторис до замовлення;
   - перелік пов'язаних послуг.
3. Перевіряється, чи потрібні немедичні послуги:
   - travel;
   - accommodation;
   - VIP / concierge;
   - support logistics.

### Фаза 4. Планування та підготовка

1. План лікування узгоджується з клієнтом.
2. За потреби вносяться корекції.
3. Після погодження план фіналізується.
4. Паралельно виконуються:
   - бронювання та підтвердження медичних термінів;
   - бронювання та підтвердження немедичних послуг;
   - призначення перекладача;
   - briefing перекладача;
   - збір і відправка підготовчих документів клієнту.
5. До старту виконання процес повинен мати повний пакет валідних даних.

### Фаза 5. Виконання програми

1. Клієнт прибуває.
2. Виконується медичний план.
3. Виконуються немедичні сервіси.
4. Надається перекладацький або супровідний сервіс.
5. У ході виконання фіксуються:
   - зміни плану;
   - факти виконаних послуг;
   - проблеми, відхилення, додаткові задачі;
   - документи та результати.

### Фаза 6. Завершення та post-care

1. Кейс переходить у локальне завершення лікування.
2. Клієнту передаються:
   - результати;
   - Arztbriefe;
   - підсумкові документи;
   - переклади, якщо вони потрібні.
3. Виставляються фінальні або проміжні рахунки.
4. Запускаються follow-up активності:
   - за лікарськими призначеннями;
   - через 1 тиждень;
   - через 1 місяць;
   - через 6 місяців;
   - за 1 місяць до завершення пакета.

## 4. Gateway rules

Ключові точки рішення:

- `Lead or Customer?`
- `Compliance completed?`
- `Lead qualified?`
- `Documents signed?`
- `Quote paid when required?`
- `Data valid?`
- `Debt exists?`
- `Billing approval granted?`
- `Order covered by existing package?`
- `Non-medical services needed?`
- `Plan correction needed?`

## 5. Ключові бізнес-правила

- Без завершеного compliance кейс не переходить у повне виконання.
- Без валідних ідентифікаційних, договірних і consent-документів виконання не стартує.
- Договір, замовлення і, за потреби, оплата кошторису є воротами запуску.
- Немедичні послуги йдуть окремою гілкою, але синхронізуються з медичним планом.
- Кожен активний кейс має відповідального `Patient Manager`.
- Якщо lead не кваліфікований або угода не відбулась, дані переходять у керований сценарій архівації або видалення.
- Якщо у клієнта є борг, до нового виконання має застосовуватись окремий debt-management workflow.

## 6. Наслідки для системи

Система повинна підтримувати:

- lifecycle `Lead -> Customer`;
- lifecycle `Order -> Execution -> Closure -> Follow-up`;
- автоматизовані checklist і reminders;
- документальну готовність кейсу;
- паралельне планування медичних і немедичних послуг;
- повний audit trail по переходах станів.
